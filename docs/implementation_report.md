# Отчёт о реализации тестового задания

Дата сверки с ТЗ: 23 марта 2026.

## 1. Модель и бизнес-ограничения (п.1 ТЗ)

### Заказы
- В системе присутствуют **уже созданные** заказы. Создание новых заказов не требуется.
- Заказ хранит:
  - `total_amount` — сумма заказа;
  - `payment_status` — вычисляемый статус оплаты: `unpaid` / `partially_paid` / `paid`.

### Платежи
- Платёж связан с заказом (`order_id`) и хранит:
  - `amount` — сумму;
  - `payment_type` — тип: `cash` / `acquiring`;
  - `status` — состояние платежа: `pending` / `succeeded` / `failed` / `refunded`.
- Модель платежа единая для всех типов: различия — на уровне `payment_type` и use-case логики.

### Операции
- **Депозит**:
  - `cash` — создаётся сразу со статусом `succeeded`;
  - `acquiring` — создаётся со статусом `pending` + сохраняется `bank_payment_id`, полученный от банка.
- **Возврат**:
  - доступен только для `succeeded` платежа;
  - переводит платеж в `refunded` и пересчитывает статус оплаты заказа.

### Ограничение суммы платежей
- Контроль реализован в `app/application/use_cases/payments.py::deposit_payment`.
- В лимит суммы заказа входят платежи со статусами `pending` и `succeeded` (резерв).
- `failed` и `refunded` в лимит не входят.

## 2. Интеграция с внешним API банка (п.2 ТЗ)

- Реализован клиент `app/infra/bank/client.py`:
  - `acquiring_start(order_id, amount)` — POST запрос на `/acquiring_start`, обработка ошибок сети/5xx/ошибки в payload;
  - `acquiring_check(bank_payment_id)` — POST запрос на `/acquiring_check`, обработка ошибок сети/404/5xx.
- Ошибки внешней системы пробрасываются как доменное исключение `ExternalServiceError`.
- Состояние платежа в банке хранится в полях `bank_payment_id`, `bank_status`, `bank_paid_at`.

### Неожиданные изменения статуса в банке
- Предусмотрен отдельный use-case `sync_acquiring_payment`, который:
  - запрашивает состояние у банка;
  - обновляет поля банка;
  - при `paid` переводит платеж в `succeeded` и пересчитывает статус заказа;
  - при `failed` переводит платеж в `failed`.

## 3. Операции и REST API (п.3 ТЗ)

Реализованы ручки FastAPI:

- `POST /orders/{order_id}/payments` — депозит
  - валидирует входные данные;
  - возвращает созданный платеж.
- `POST /payments/{payment_id}/refund` — возврат
  - корректно формирует `204 No Content`;
  - обрабатывает конфликт/валидацию/внешние ошибки.
- `POST /payments/{payment_id}/sync` — синхронизация эквайринга
  - обновляет состояние эквайрингового платежа по данным банка.

Ошибки внутренних операций маппятся в HTTP-ответы через `HTTPException`.

## 4. Хранение данных и инфраструктура (п.4 ТЗ)

- Используется PostgreSQL в Docker.
- Миграции: Alembic (`alembic/`).
- Сидирование: при старте API создаются тестовые заказы, если таблица `orders` пустая.
- Схема БД описана в `docs/db_schema.md`.
